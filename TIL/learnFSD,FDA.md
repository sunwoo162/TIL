### FSD와 FDA
- FDA
	- 'Feature Driven Architecture'
	- 좋은 코드의 조건
	- 코드의 역할과 책임을 나누는 원리
	- Discoverability
		- 프로젝트 규마가 커질 수록 기능을 찾기 어려워짐
	- Work parallelisation
		- 여러 명이 다른 기능을 동시에 작업하면 어려워짐
	- Controlling shared absractions
		- 공통적으로 사용되는 코드는 찾기 쉬운 위치에 직관적인 이름으로 작성되어있다(접근성 높음)
	- 모놀리스
		- 도메인이 아닌 기술적인 기능으로만 일차원 적으로 배치해놓는 것(어디가 어떤 기능인지 파악하기 어려움)
	- 피쳐 단위로 코드를 나누기
		- 잘게 나눔 약간 프로필 이렇게 하고 그 안에 기능 사진 이런식으로 나누는거
	- 캡슐화
		- 폴더 내부를 숨겨서 외부에서 내부가 뭘 하는지 모르게 함
		- 왜 쓸까 
			- 내부 구조 변경이 자유롭다
			- 협업에 유리하다(다른 개발자는 index.js만 보면 됨
	- Explicit Sharing
		- 코드에서 버그가 나오는 것을 테스트코드로 100% 해결 불가
		- 타입스크립트와 같이 정적 타이핑을 이용해야함
	- co-location
		- 한 피쳐 내부에 쓰이는 도메인 코드가 여러 곳으로 분류되있으면 찾기 어렵기 때문에 한곳에 위치시킨다.
	- de-coupling isolation
		- 피쳐 코드 내부에서 사용하는 코드들은 다른 피쳐 모듈을 의존하면 안됨(변경 영향 최소화)
		- page 코드는 다른 page를 의존하면 안됨
		- shared 코드가 다른 타입을 의존하면 안됨(어디서 뭐가 터졌는지 확인 ㄱㄴ)

- FSD
	- 'Feature Sliced Design'
	- Layers, Slices, Segments 로 구성되어있음
	- Layers
		- 추상화 레벨과 비지니스 레벨
	- slices
		- FSD의 두번째 계층으로 이름이 가져야 할 특별한 규약은 없다
		- 비지니스 도메인으로 슬라이스 이름이 정해짐
	- Segments
		- FSD의 마지막 계층 요소 한 도메인 안에서 기술적인 결과를 달성하기 위해 작성함
		- 이름은 ui, model, lib, api등이 될 수 있다.
			- ui : UI 컴포넌트
			- model : 비지니스 로직, data aggregation 함수들
			- lib : infra structural code
			- api : backend api를 호출하기 위한 코드
	- Layers 구성요소
		- app
			- 앱의 설정을 맡는 레이어로 여러 모듈에서 재사용하지 않으나 앱에 고루 영향을 미치는 모듈들을 저장하는 곳
		- pages
			- 하위 레이어들을 조합하여 완전한 기능을 제공하는 레이어로 앱에서 각 라우터에 해당하는 페이지들을 작성하는 곳
		- widgets
			- 위젯은 하위의 레이어들을 이용해 특정 피쳐에서 사용할 수 있는 UI 블록들을 만드느 곳이다. 
			- 선택적 레이어로 불필요하다면 생략 ㄱㄴ
		- features
			- 버튼 클릭과 같은 유저의 인터렉션과 관련된 로직, 특정 비즈니스 로직이 담겨있는 레이어
		- entities
			- 특정 도메인의 모델을 정의하고 그와 연관된 api를 호출하는 api 어뎁터를 작성함
